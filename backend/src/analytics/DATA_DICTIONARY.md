# Analytics Data Dictionary

This document is the single source of truth for every column on every persisted
analytics entity: what it means, its unit/format, and how it gets populated —
via a **live hook** (written as part of a request/event, in near real time) or
a **nightly job** (written by a scheduled batch process on a rollup table).

See also [EVENT_TAXONOMY.md](./EVENT_TAXONOMY.md) for the naming convention and
payload shape of individual `eventType` values recorded on `AnalyticsEvent`.

## Entities

| Entity | Table | Population |
|--------|-------|------------|
| [`AnalyticsEvent`](#analyticsevent) | `analytics_events` | Live hook |
| [`RetentionCohort`](#retentioncohort) | `retention_cohorts` | Nightly job (see [note](#retentioncohort)) |

### `AnalyticsEvent`

File: [`entities/analytics-event.entity.ts`](./entities/analytics-event.entity.ts)

The raw, append-only event log. One row per tracked occurrence (a puzzle
attempt, a login, an XP award, etc.). Every other analytics feature —
funnels, churn risk, and eventually retention rollups — is derived from this
table, either at query time or by a batch job that reads it.

| Column | Type | Nullable | Description |
|--------|------|----------|--------------|
| `id` | `uuid` | No | Primary key. Generated by the database. |
| `eventType` | `varchar(100)` | No | Event name, e.g. `puzzle_attempted`, `streak_broken`. Must follow the `noun_pastTenseVerb` convention documented in [EVENT_TAXONOMY.md](./EVENT_TAXONOMY.md). |
| `userId` | `varchar(100)` | No | Identifies the user the event belongs to. |
| `entityId` | `varchar(100)` | No | Identifier of the domain object the event is about (e.g. a puzzle ID, quest ID). Populated by the emitting listener/provider; its meaning is scoped by `eventType`. |
| `payload` | `json` | Yes | Free-form metadata specific to `eventType`. Shape documented per event in [EVENT_TAXONOMY.md](./EVENT_TAXONOMY.md) (e.g. `{ puzzleId, difficulty, isCorrect, timeSpent }` for `puzzle_attempted`). |
| `timestamp` | `timestamptz` | No | When the event occurred. Set automatically on insert (`@CreateDateColumn`); not client-supplied. |

Indexes: `(timestamp, userId)` for range scans over a reporting window
grouped by user; `(userId, timestamp)` for a single user's history in time
order.

**Population — live hook, two write paths:**

1. **Domain event listener.** [`UsersAnalyticsListener`](./listeners/users-analytics.listener.ts)
   subscribes to the in-process `xp_awarded` and `user_leveled_up` events
   (emitted elsewhere in the app, e.g. by the progress module) via
   `@nestjs/event-emitter` and writes one `AnalyticsEvent` row per occurrence,
   immediately.
2. **Direct tracking API.** [`TrackEventProvider`](./providers/track-event.provider.ts)
   is called from `POST /analytics/track` (see
   [`AnalyticsController`](./controllers/analytics.controller.ts)) and by any
   other module that wants to record an event (e.g. progress, streak, quests
   providers calling `TrackEventProvider.track()`). Rows are buffered
   in-memory and written in a batch — whichever comes first: the buffer
   reaching 50 events, or a 5-second flush interval. This means a given row
   may land in `analytics_events` up to ~5 seconds after the triggering
   action, not necessarily on the same transaction as it.

There is no nightly job that writes to this table — it is only ever read by
rollup/aggregation logic, never rewritten.

### `RetentionCohort`

File: [`entities/retention-cohort.entity.ts`](./entities/retention-cohort.entity.ts)

A pre-aggregated rollup table: one row per acquisition cohort (users who
first appeared on `cohortDate`), recording how many of those users returned
on day 1, 7, and 30. Exists so [`GetRetentionCurveProvider`](./providers/get-retention-curve.provider.ts)
can serve `GET /analytics/users/retention` from an indexed rollup instead of
scanning raw `analytics_events` rows at request time.

| Column | Type | Nullable | Description |
|--------|------|----------|--------------|
| `id` | `uuid` | No | Primary key. Generated by the database. |
| `cohortDate` | `date` | No | The acquisition date this cohort is keyed on (`YYYY-MM-DD`), i.e. the day these users first appeared. Indexed for range scans by date. |
| `cohortSize` | `int` | No (default `0`) | Total number of users in the cohort on `cohortDate`. |
| `retainedDay1` | `int` | No (default `0`) | Count of cohort users who returned exactly 1 day after `cohortDate`. |
| `retainedDay7` | `int` | No (default `0`) | Count of cohort users who returned exactly 7 days after `cohortDate`. |
| `retainedDay30` | `int` | No (default `0`) | Count of cohort users who returned exactly 30 days after `cohortDate`. |
| `createdAt` | `timestamptz` | No | Row creation timestamp, set automatically. |
| `updatedAt` | `timestamptz` | No | Row last-updated timestamp, set automatically on save. |

`day1RetentionPct` / `day7RetentionPct` / `day30RetentionPct` seen in API
responses (`RetentionDataPoint`) are **not** stored columns — they are
computed on read as `retainedDayN / cohortSize`, returned as `null` when
`cohortSize` is `0` rather than dividing by zero.

**Population — intended to be a nightly job; not yet implemented.**
The entity is documented and designed for a nightly aggregation job to
populate it end-to-end (compute cohort membership and day 1/7/30 returns
from `analytics_events`, then upsert one row per `cohortDate`). As of this
writing, **no such job exists in the codebase** — nothing currently writes
to `retention_cohorts`, so `GetRetentionCurveProvider` will return empty
results (`{ data: [], total: 0 }`) against an unpopulated table. This is a
gap to close in a follow-up issue, not something this document works around.

## Derived metrics (not persisted)

The following response shapes are computed on the fly from the entities
above and returned by API endpoints. They are not stored anywhere and have
no "written by" answer — they exist only for the duration of a request —
but are included here since dashboards consume them the same way they'd
consume a table.

| Result type | Endpoint | Computed from |
|--------------|----------|----------------|
| `FunnelResult` ([`interfaces/funnel-result.interface.ts`](./interfaces/funnel-result.interface.ts)) | `GET /analytics/funnel/onboarding` | `COUNT(*)` per onboarding `eventType` over `analytics_events`, in the fixed stage order defined in [`GetOnboardingFunnelProvider`](./providers/get-onboarding-funnel.provider.ts). |
| `RetentionDataPoint` ([`dtos/analytics-metric-result.dto.ts`](./dtos/analytics-metric-result.dto.ts)) | `GET /analytics/users/retention` | Read directly from `RetentionCohort` rows; percentages derived as described above. |
| `ChurnRiskDataPoint` ([`dtos/analytics-metric-result.dto.ts`](./dtos/analytics-metric-result.dto.ts)) | `GET /analytics/users/churn-risk` | Per-user activity counts bucketed by day/week/month over `analytics_events` (restricted to `ACTIVITY_EVENT_TYPES` in [`GetChurnRiskProvider`](./providers/get-churn-risk.provider.ts)), scored as a z-score drop against the user's own baseline. |

## Adding a new entity or column

1. Add the column to the entity class and, if the schema changes, a
   migration.
2. Document it in this file under the appropriate entity's table, including
   whether it's populated by a live hook or a nightly job.
3. If it's a new raw event type rather than a new column, add it to
   [EVENT_TAXONOMY.md](./EVENT_TAXONOMY.md) instead — this file documents
   entities/columns, that one documents individual event payloads.
